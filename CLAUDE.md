# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

A framework-free zsh + git + starship dotfiles repo. Plain config files, organized into modules, deployed to `$HOME` by symlink. No oh-my-zsh / prezto / plugin manager.

`CHEATSHEET.md` is the human-facing quick reference (new-machine setup + the keymaps/aliases worth remembering). It is intentionally **not** exhaustive. When a change adds or renames something a person would reach for from memory ‚Äî a new keymap domain, a changed leader binding, an install step, a headline alias ‚Äî update the relevant `CHEATSHEET.md` section in the same change. Don't mirror every binding there; if it's only discoverable via which-key or the config, it doesn't need a cheatsheet line.

## Commands

- `./install.sh` ‚Äî runs `brew bundle`, seeds untracked local files, then symlinks everything into `$HOME`. Idempotent: skips already-correct links, backs up anything in the way to `<file>.bak.<timestamp>` (gitignored).
- `exec zsh -l` ‚Äî reload a full login shell to test changes (there is no build/test suite).
- After editing only an interactive module: `source ~/.zshrc`.

## Load order (the core mental model)

zsh sources files in a fixed sequence, and this repo's structure mirrors it. Putting code in the wrong file is the most common mistake:

1. **`zsh/zshenv`** ‚Üí `~/.zshenv` ‚Äî every zsh (scripts, ssh commands, non-TTY included). Keep it fast and silent: env vars only, nothing that prints.
2. **`zsh/zprofile`** ‚Üí `~/.zprofile` ‚Äî login shells, *after* macOS's `/etc/zprofile` path_helper reorders PATH. This is the only correct place for PATH-mutating tool init (Homebrew `shellenv`, asdf shims) so those tools win.
3. **`zsh/zshrc`** ‚Üí `~/.zshrc` ‚Äî interactive shells only. A thin loader that sources `conf.d/*.zsh` in numeric order, then alias modules, then plugins.

Within `zshrc`, **interactive plugins must load last and in a specific order**: zsh-autosuggestions ‚Üí zsh-syntax-highlighting ‚Üí zsh-history-substring-search (the last two depend on highlighting being loaded, and the history-search keybindings are bound after sourcing because the widgets must exist first).

`conf.d/` numeric prefixes (`00-options`, `10-completion`, `20-keybindings`, `30-prompt`, `40-tools`, `50-aliases`) encode ordering dependencies ‚Äî e.g. `00-options` sets `KEYTIMEOUT` before `20-keybindings` enables `bindkey -v`; fpath additions in `10-completion` precede its `compinit`.

## Key architectural decisions

- **Stable-path indirection.** `install.sh` symlinks `~/.config/zsh` ‚Üí `repo/zsh`, and `zshrc` resolves everything through `ZSH_DIR=${ZDOTDIR:-$HOME/.config/zsh}`. This makes the loader work regardless of where the repo is cloned.
- **`.local` override seam.** Each tracked file sources an untracked sibling if present, where machine-/work-specific values (secrets, TORO_* tokens, identity, internal URLs) go ‚Äî these are intentionally never committed: `~/.zshenv.local`, `~/.zprofile.local`, `~/.zshrc.local`, `~/.gitconfig.local`. The git case uses git's native `[include]` of `~/.gitconfig.local` (a later include always wins); `install.sh` seeds it from `git/gitconfig.local.example` if missing.
- **Vendored prezto aliases/functions.** `zsh/aliases/{git,ruby,rails}.zsh` are reproduced verbatim from prezto. The function-backed aliases (`gcl`, `gRb`, `gsX`, etc.) work only because their helpers are vendored under `zsh/functions/`, where **filename = function name** and each is autoloaded via `autoload -Uz "$ZSH_DIR"/functions/*(:t)`. To add an alias module, drop a `*.zsh` in `aliases/` (auto-sourced); to add a helper, add a file in `functions/` named for the function.
- **Deliberate non-aliasing.** `cat`‚Üí`bat` and `ls`‚Üí`eza` are aliased (bat detects non-TTY and acts like cat, so it's pipe-safe), but `find`/`grep` are deliberately *not* aliased to `fd`/`rg` because their flags differ and would surprise scripts.

## Tooling

Dependencies are pinned in `Brewfile`. Runtime versions (ruby, node) are managed by **asdf** (Go rewrite, 0.16+: shims on PATH, no `asdf.sh`) via `~/.tool-versions`. Git is configured for SSH commit/tag signing and git-delta as the pager.

The command line is floated by **noice.nvim** (`lua/plugins/noice.lua`, with nui + nvim-notify), and `options.lua` sets `cmdheight = 0` to reclaim the bottom line ‚Äî so `:`/search appear in a centered popup and messages route through nvim-notify. Note: noice only initializes when a UI is attached, so it no-ops (and `:Noice` won't exist) under `nvim --headless` ‚Äî verify cmdline/message behavior in a real session.

`nvim/` is a modular neovim config managed by **lazy.nvim** ‚Äî deliberately *not* a distro (LazyVim/NvChad), so every plugin is an explicit, editable spec. `init.lua` sets the leader (`,`) and loads `lua/config/{options,keymaps,autocmds,lazy}.lua`; `lua/config/lazy.lua` then `import`s every file under `lua/plugins/`, one file per concern. To add/remove a plugin, add/delete a spec file (or a table in one) ‚Äî no central registry. The whole `nvim/` dir is symlinked to `~/.config/nvim`. LSP servers and formatters install on demand via **Mason** (`:Mason`), so a fresh machine self-provisions on first launch. Treesitter is on the **`main`** branch (the `master` branch is frozen and crashes on Neovim 0.11+/0.12 with a `range (nil)` injection error). The `main` API differs: no `nvim-treesitter.configs`, parsers are installed via `require('nvim-treesitter').install()` and highlighting is started per buffer in a `FileType` autocmd, and it **compiles parsers with the `tree-sitter` CLI** (the `tree-sitter-cli` Brewfile formula ‚Äî not the library-only `tree-sitter` formula). The textobjects plugin is also on `main` (manual keymaps); endwise is `tpope/vim-endwise` since the treesitter-endwise plugin needed the dead `configs` API. Light/dark follows the macOS appearance event-driven via **dark-notify** (`cormacrelf/dark-notify`, the same brew binary tmux uses): `lua/plugins/colorscheme.lua` flips the kanagawa variant (wave/lotus) over a pipe ‚Äî no polling. Tuned for the Toro stack: Ruby/Rails + GraphQL + Postgres backend, TypeScript/React/Vite frontend.

`ghostty/config` sets the terminal theme (light/dark Kanagawa) and Lilex Nerd Font, linked into `~/.config/` by `install.sh` like the other configs.

`tmux/tmux.conf` (symlinked to `~/.config/tmux/`) uses the default `C-b` prefix and manages plugins with **TPM**, which `install.sh` bootstraps (clones to `~/.config/tmux/plugins/tpm` and runs `install_plugins`) the same way it bootstraps lazy.nvim. Plugins: tmux-sensible, tmux-yank, tmux-fzf, resurrect + continuum (auto save/restore), vim-tmux-navigator, and tmux-dark-notify. That last one is paired with the `christoomey/vim-tmux-navigator` nvim plugin (`nvim/lua/plugins/tmux.lua`) so `C-h/j/k/l` moves seamlessly across tmux panes and nvim splits ‚Äî which is why nvim's `keymaps.lua` does **not** map `C-hjkl` itself. The status bar is hand-rolled (no theme plugin) and **follows the macOS appearance** like Ghostty/nvim, via the **tmux-dark-notify** plugin: it runs the `dark-notify` brew binary (event-driven ‚Äî no polling) and sources `tmux/themes/wave.conf` (dark) or `lotus.conf` (light) the instant the system flips, symlinking the active file to `~/.local/state/tmux/tmux-dark-notify-theme.conf`. `tmux.conf` sources that symlink once after TPM init as a startup fallback (the plugin re-sources on the next change); the two `@dark-notify-theme-path-{dark,light}` options point it at the theme files. Colors/segment formats live in the theme files; only structure (position, lengths) lives in `tmux.conf`. Each theme's `status-right` prepends a now-playing segment: `tmux/scripts/now-playing.sh` (invoked via `#()` every `status-interval`) prints the current Apple Music track via AppleScript, guarded by `application "Music" is running` so it never launches Music and emits nothing when idle ‚Äî it talks to Music.app directly because macOS 15.4+ locked down the system-wide MediaRemote framework that universal tools like `nowplaying-cli` read.

Parallel agent work is driven by **workmux** (`raine/workmux`), a Rust CLI that pairs one git worktree per task with a tmux window running Claude Code plus a shell (`wm add <name>` / `wm merge`). It is **not** vendored or installed via brew/crates.io: the dotfiles track only the global config (`workmux/config.yaml` ‚Üí symlinked to `~/.config/workmux/`) and build the binary **from a personal fork**. `workmux/install.sh` (run by the top-level `install.sh` after `brew bundle` provides the `rust` toolchain) clones `jdeff/workmux` to `~/src/workmux` ‚Äî with an `upstream` remote pointing at `raine/workmux` ‚Äî and `cargo install --root ~/.local`s it to `~/.local/bin/workmux` (already on PATH via `zprofile`). Rebuild after pulling with the `workmux-update` shell function (next to the `wm` alias in `conf.d/50-aliases.zsh`) or by re-running `workmux/install.sh`. The status integration is deliberate: `status_format: false` in the config stops workmux from rewriting `window-status-format`, so the hand-rolled status bar keeps control ‚Äî workmux just sets the `@workmux_status` window user-option, which `tmux/themes/{wave,lotus}.conf` render via `#{?@workmux_status,‚Ä¶}`. Status glyphs are **bare** Nerd Font characters in `config.yaml` (no color codes) so the theme files color them per light/dark appearance, like every other segment. Full details and the upstream-sync/uninstall steps live in `workmux/README.md`.
